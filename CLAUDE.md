# app-manifest-cli

CLI tool (`am`) that generates **Application Manifest v2** (AMv2) JSON files in CycloneDX 1.6 BOM format. Used in CI/CD pipelines to describe which Docker images, Helm charts, and other artifacts make up a deployable application.

## Commands

```bash
uv run am <command>                             # run CLI
uv run pytest tests/                            # run all tests
uv run pytest tests/test_dd_converter.py        # run specific test file
uv run pytest tests/ -k "TestJaeger"            # run by class/keyword
```

CLI aliases: `c` = component, `gen` = generate, `f` = fetch, `v` = validate, `cv` = convert, `i` = info.

## Domain glossary

| Term | Meaning |
|------|---------|
| **AMv2** | Application Manifest v2 — the CycloneDX 1.6 BOM JSON this tool produces |
| **DD** | Deployment Descriptor — legacy JSON format (services + charts), input for `convert --to-am` |
| **mini-manifest** | Per-component CycloneDX JSON generated by `component` or `fetch`; assembled into the final AMv2 by `generate` |
| **Build Config** | YAML file passed via `--config`; lists components and their `dependsOn` relationships |
| **regdef** | Registry Definition YAML (`--registry-def`); maps a registry name to Docker/Helm URLs |
| **standalone-runnable** | The application entry-point component in AMv2; always present, built from Build Config (no mini-manifest) |
| **app-chart** | The umbrella Helm chart (DD `type: app-chart`) that contains service charts as nested components |
| **service chart** | Per-service Helm chart (nested inside app-chart); linked to a Docker image via `artifactMappings` |
| **sub-chart** | A Helm chart that is a `dependsOn` of another Helm chart in Build Config; placed inside `components[]` of the parent, not at top level |
| **bom-ref** | Unique component identifier in the format `name:uuid4`, generated by `_make_bom_ref(name)`. Regenerated on every run — never store or compare across runs. |
| **artifactMappings** | A `CdxProperty` on a Helm chart component that maps `{docker_bom_ref: {"valuesPathPrefix": "..."}}` |

## Architecture

### Pipeline: normal flow (`component` → `fetch` → `generate`)

```
CI metadata JSON  ──► component  ──► mini-manifest JSON ──┐
Helm/Docker refs  ──► fetch      ──► mini-manifest JSON ──┼──► generate ──► AMv2 JSON
Build Config YAML ─────────────────────────────────────────┘
```

### Pipeline: conversion flow (`convert`)

```
DD JSON + Build Config + regdef ──► convert --to-am ──► AMv2 JSON
AMv2 JSON + regdef              ──► convert --to-dd ──► DD JSON
```

DD→AMv2 conversion is **isolated** from the normal pipeline — it lives entirely in `services/dd_converter.py` and does not touch `manifest_builder`, `artifact_fetcher`, or `component_builder`.

## Services overview

| File | Responsibility |
|------|---------------|
| `cli.py` | Click commands, argument parsing, error formatting |
| `services/manifest_builder.py` | Assembles mini-manifests + Build Config → final `CycloneDxBom` |
| `services/component_builder.py` | Converts `ComponentMetadata` (CI JSON) → mini-manifest `CycloneDxBom` |
| `services/artifact_fetcher.py` | Helm: runs `helm pull`, extracts Chart.yaml + nested files. Docker: creates minimal mini-manifest from `reference` (no pull, no hash) |
| `services/helm_fetcher.py` | Backwards-compat re-export from `artifact_fetcher` — do not import directly |
| `services/dd_converter.py` | DD ↔ AMv2 conversion logic; also handles ZIP extraction for values.schema.json and resource-profiles |
| `services/purl.py` | Generates and parses PURLs for Docker and Helm artifacts |
| `services/validator.py` | Validates manifest JSON against `schemas/application-manifest.schema.json` using `jsonschema` |
| `services/config_loader.py` | Loads and validates Build Config YAML → `BuildConfig` |
| `services/metadata_loader.py` | Loads CI metadata JSON → `ComponentMetadata`; also loads mini-manifests from files/dirs |
| `services/regdef_loader.py` | Loads Registry Definition YAML → `RegistryDefinition` |

## Model map

### CI metadata input (`models/metadata.py`)
```
ComponentMetadata                        ← CI JSON passed to `component`
  name: str                              ← must match name in Build Config for generate to link them
  type: str                              ← "container" (Docker) or "application" (Helm)
  mime_type: str    (alias: mime-type)
  group: str | None                      ← Docker namespace / org
  version: str | None                    ← Docker tag or Helm chart version
  app_version: str | None               ← Helm appVersion; takes priority over version in output
  hashes: list[HashEntry]
  reference: str | None                  ← full artifact reference for PURL
  components: list[MetadataNestedComponent]   ← pre-built values.schema.json, resource-profiles from CI
```

### Build Config (`models/config.py`)
```
BuildConfig
  application_name: str    (alias: applicationName)
  application_version: str (alias: applicationVersion)
  components: list[ComponentConfig]
    └── ComponentConfig
          name: str
          mime_type: MimeType  (alias: mimeType)
          reference: str | None          ← if set, fetch will pull this artifact
          depends_on: list[DependencyConfig]  (alias: dependsOn)
            └── DependencyConfig
                  name: str
                  mime_type: MimeType    (alias: mimeType)
                  values_path_prefix: str | None  (alias: valuesPathPrefix)
```

### Registry Definition (`models/regdef.py`)
```
RegistryDefinition
  name: str                              ← used as registry_name in PURLs (e.g. "qubership", "sandbox")
  docker_config: DockerConfig | None     (alias: dockerConfig)
    └── group_uri: str | None            ← Docker registry host (e.g. "ghcr.io")
        group_name: str | None           ← namespace/org prefix (e.g. "netcracker")
        snapshot_uri, staging_uri, release_uri: str | None   ← alternate registry URIs
  helm_app_config: HelmAppConfig | None  (alias: helmAppConfig)
    └── repository_domain_name: str | None  ← Helm registry base URL (e.g. "oci://registry.qubership.org")
        helm_group_repo_name: str | None    ← path prefix (e.g. "helm-group")
  github_release_config: GitHubReleaseConfig | None  (alias: githubReleaseConfig)
```

### AMv2 output (`models/cyclonedx.py`)
```
CycloneDxBom
  ├── CdxMetadata
  │     ├── CdxMetadataComponent   ← describes the application itself (name, version)
  │     └── CdxToolsWrapper → CdxTool
  ├── CdxComponent  (recursive — components can contain components)
  │     ├── CdxProperty            ← e.g. isLibrary=False, artifactMappings
  │     ├── CdxHash                ← SHA-256 digest
  │     └── CdxDataEntry → CdxDataContents → CdxAttachment  ← base64-embedded files
  └── CdxDependency  (ref → dependsOn list of bom-refs)
```

### DD models (`models/dd.py`)
```
DeploymentDescriptor
  ├── DdService  (image_name, full_image_name, image_type, service_name, docker_tag, ...)
  └── DdChart    (helm_chart_name, helm_chart_version, full_chart_name, ...)
```

`image_type="service"` → service has an associated Helm chart (`service_name`).
`image_type="image"` → standalone Docker image, no Helm chart.

## Key behavioral rules

**One name, two mimeTypes (common pattern):**
- A single `name` can appear multiple times in Build Config with different `mimeType` values. Example: `qubership-jaeger` as both `standalone-runnable` and `helm.chart`.
- The lookup key is always `(name, mime_type)` — never name alone. Two components with the same name but different mimeType are completely independent entries.

**Component naming:**
- In mini-manifests from `component`: `name` is always taken from the CI metadata JSON.
- In mini-manifests from `fetch`: `name` is always taken from the **Build Config** (not from Chart.yaml or the image reference). This ensures `generate` can match them.
- `generate` matches mini-manifests to config by `(name, mime_type)` tuple — both must match.

**`generate` positional arguments:**
- Mini-manifest inputs can be individual JSON files, directories (all `*.json` inside are loaded), or a mix of both.
- Within a directory, files are loaded in **alphabetical order by filename**.
- Parent directories of the output file are created automatically.

**`generate` index collision (silent):**
- If two files contain a component with the same `(name, mime-type)`, the second **silently overwrites** the first — no warning, no error.
- Order determines the winner: alphabetical within a dir; left-to-right across CLI args.

**`generate` does NOT re-generate PURLs:**
- PURLs are copied from mini-manifests as-is. `generate` never touches them.
- Consequence: `--registry-def` must be passed consistently to both `component` and `fetch`, not just to `generate`.

**Helm version resolution (in component_builder):**
- `appVersion` from CI metadata → used as the component `version` in the output.
- Falls back to `version` if `appVersion` is absent.

**fetch behavior:**
- Helm: runs `helm pull` (requires `helm` CLI on PATH **and** `helm registry login` done beforehand), computes SHA-256 of the `.tgz`, extracts `values.schema.json` and `resource-profiles/*.yaml` as nested components.
- Docker: creates a minimal mini-manifest from `reference` — **no image pull, no hash**.
- Docker components **without** `reference` in Build Config are **silently skipped** by `fetch` — they must come from `component`.

**Sub-charts (manifest_builder):**
- If a Helm chart A has a Helm chart B in `dependsOn`, then B is a **sub-chart**.
- Sub-charts are embedded inside `components[]` of A, not placed at the top level of the manifest.
- Sub-charts still get their own entry in `dependencies[]`.
- Sub-charts are built from config only — they have **no version, no purl, no hashes** in the final manifest.

**artifactMappings property:**
- Added to Helm chart components when a docker dependency has `valuesPathPrefix` set.
- Key is the **bom-ref** of the docker image (not its name).
- Value is `{"valuesPathPrefix": "images.jaeger"}`.
- `valuesPathPrefix: "."` means the image maps to the root of `values.yaml`.
- Property name: `"qubership:helm.values.artifactMappings"`.

**standalone-runnable:**
- Always built from Build Config, never from a mini-manifest.
- Version comes from `--version` CLI flag, or `applicationVersion` from Build Config.

**Warnings:**
- All warnings go to **stderr**, not stdout.
- `generate` and `convert` return `(bom, warnings)` — warnings printed after building.
- Missing mini-manifest → warning + component skipped (not an error).

**JSON Schema validation:**
- Schema lives at `src/app_manifest/schemas/application-manifest.schema.json`.
- `validate` command and `generate --validate` both use `jsonschema.Draft7Validator`.

## Pydantic conventions

All models follow the same pattern:

```python
# YAML/JSON uses camelCase; Python uses snake_case
mime_type: MimeType = Field(alias="mimeType")

# Always set on every model — allows both Python name and alias in tests/code
model_config = {"populate_by_name": True}
```

**Serialization** always uses:
```python
obj.model_dump(by_alias=True, exclude_none=True)   # AMv2 output (omit nulls)
obj.model_dump(by_alias=True, exclude_none=False)   # DD output (preserve nulls)
```

`CdxComponent` uses `validation_alias` / `serialization_alias` (not `alias`) because its field names contain hyphens (`bom-ref`, `mime-type`) that are invalid as Python identifiers.

## MimeType reference

| Enum value | String |
|-----------|--------|
| `STANDALONE_RUNNABLE` | `application/vnd.nc.standalone-runnable` |
| `DOCKER_IMAGE` | `application/vnd.docker.image` |
| `HELM_CHART` | `application/vnd.nc.helm.chart` |
| `SMARTPLUG` | `application/vnd.nc.smartplug` |
| `SAMPLEREPO` | `application/vnd.nc.samplerepo` |
| `CDN` | `application/vnd.nc.cdn` |
| `CRD` | `application/vnd.nc.crd` |
| `JOB` | `application/vnd.nc.job` |

Additional MIME types used internally (not in the enum):
- `application/vnd.nc.helm.values.schema` — embedded `values.schema.json`
- `application/vnd.nc.resource-profile-baseline` — embedded resource profiles
- `application/vnd.nc.application` — application entry in `metadata.component`

`application/vnd.qubership.*` variants (e.g. `application/vnd.qubership.helm.chart`) are treated identically to their `vnd.nc.*` counterparts throughout the codebase.

## PURL formats

```
Docker:  pkg:docker/{namespace}/{name}@{tag}?registry_name={regdef_name}
Helm:    pkg:helm/{name}@{version}?registry_name={regdef_name}
         pkg:helm/{namespace}/{name}@{version}?registry_name={regdef_name}
```

`registry_name` resolves via regdef: if `registry_host` matches `dockerConfig.groupUri` (and `namespace` matches `groupName`) → `regdef.name`; otherwise the host itself is used as fallback.

PURL generation: `services/purl.py`. PURL → artifact-ref reverse: `services/dd_converter.py`.
